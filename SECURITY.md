# Security policy

## Supported versions

The plugin is published to the Gradle Plugin Portal as
`io.github.vedanthvdev.affectedtests`. Security fixes are issued
against the latest minor line on `master` only. Older minor lines
are not back-patched — adopters should track `master`'s latest
release.

| Version | Security fixes |
| --- | --- |
| Latest 2.x | Yes |
| Older 2.x | No — upgrade to latest 2.x |
| 1.x | No — superseded by 2.x |

## Verifying artifact provenance

Releases published from this repo's CI **after the build
provenance pipeline landed** (issue #83 — first attested release
will be the next tag cut from `master`) ship with a [SLSA build
provenance attestation](https://docs.github.com/en/actions/security-for-github-actions/using-artifact-attestations/using-artifact-attestations-to-establish-provenance-for-builds)
generated by `release.yml` and recorded in Sigstore's public
transparency log (Rekor). The attestation binds the published
JAR to:

- The exact commit SHA on `master` it was built from.
- The exact `release.yml` workflow file (at its commit SHA) that
  built it.
- The GitHub Actions workflow run that produced it.

This closes the gap that a personal-account namespace
(`io.github.vedanthvdev.*`) would otherwise leave open: a
consumer can prove cryptographically that any given JAR came
from this repo's CI rather than a compromised credential or a
laptop publish. **Older releases will return "no attestation
found" and that is expected.** Track the [Releases](https://github.com/vedanthvdev/affected-tests/releases)
page for the first attested version.

### What gets attested vs what gets resolved

The Plugin Portal hosts two artifacts per release:

| Coordinate | Role | Attested? |
| --- | --- | --- |
| `io.github.vedanthvdev:affected-tests-gradle:<version>` | The implementation shadow JAR — relocated JGit, JavaParser, slf4j and the plugin's own classes | **Yes** |
| `io.github.vedanthvdev.affectedtests:io.github.vedanthvdev.affectedtests.gradle.plugin:<version>` | The plugin marker — a tiny POM stub Gradle uses to look up the implementation when you write `plugins { id 'io.github.vedanthvdev.affectedtests' ... }` | No — it carries no executable code |

The implementation JAR is the only artifact that contains code,
so it is the only one that needs cryptographic provenance. The
marker is a published-by-`com.gradle.plugin-publish` POM
indirection layer; verifying it would tell you nothing the
implementation attestation does not already say.

### Verification in two steps

1. Resolve the implementation JAR Gradle pulled from the Plugin
   Portal. When the plugin is applied via the `plugins {}` DSL,
   the JAR lands in your Gradle cache at:

   ```
   ~/.gradle/caches/modules-2/files-2.1/io.github.vedanthvdev/affected-tests-gradle/<version>/<sha1>/affected-tests-gradle-<version>.jar
   ```

   Note `io.github.vedanthvdev` (the publisher group), not
   `io.github.vedanthvdev.affectedtests` (the marker / plugin id).

2. Verify with the [GitHub CLI](https://cli.github.com/):

   ```bash
   gh attestation verify <path-to-jar> \
     --owner vedanthvdev \
     --repo vedanthvdev/affected-tests
   ```

   A successful verification prints the workflow file, commit
   SHA, and run URL the JAR was produced by. A mismatch — wrong
   owner, wrong workflow, missing attestation — exits non-zero
   so it can be wired into a CI gate without further parsing.

   Pinning `--repo` (in addition to `--owner`) hardens the check
   against a future namespace transfer. The default predicate
   type (`https://slsa.dev/provenance/v1`) is already enforced
   unless overridden — pass `--predicate-type` explicitly if you
   want to reject a future downgrade to a non-SLSA predicate.

### Wiring it into your own CI

Adopters running this plugin in a security-sensitive pipeline
can fail-closed on missing or mismatched provenance. Resolve
the implementation JAR by exact version (avoids picking up a
stale older copy if multiple versions are cached):

```bash
VERSION="<the version your build resolved>"   # e.g. 2.10.0
JAR=$(find ~/.gradle/caches/modules-2/files-2.1/io.github.vedanthvdev/affected-tests-gradle/"$VERSION" \
  -type f -name "affected-tests-gradle-${VERSION}.jar" | head -1)

gh attestation verify "$JAR" \
  --owner vedanthvdev \
  --repo vedanthvdev/affected-tests \
  || { echo "::error::affected-tests JAR failed provenance verification"; exit 1; }
```

## Reporting a vulnerability

**Do not open a public GitHub issue for a security report.**

Use GitHub's
[private vulnerability reporting](https://github.com/vedanthvdev/affected-tests/security/advisories/new)
flow on the repo's Security tab. The form routes the report
directly to the maintainers in a private advisory thread before
any code or detail becomes public.

If GitHub private reporting is not an option for you, email the
maintainer at the address listed on their GitHub profile and put
`[SECURITY]` in the subject line. Include:

- A description of the vulnerability and the threat it enables.
- A minimal reproduction (a Gradle project layout, a diff, or a
  command line is plenty).
- The affected plugin version (`./gradlew :buildEnvironment` will
  print it).

## What to expect

- **Acknowledgement** within 7 days of the report landing.
- **Triage** within 14 days — either a confirmed CVE-track issue
  with an ETA, or a reasoned explanation for why the report does
  not meet the threat model below.
- **Fix + advisory** issued together. The advisory will credit the
  reporter unless they ask to remain anonymous.

## Threat model — what this plugin treats as an untrusted boundary

The plugin sees three categories of input. The threat model
treats them differently and a security report is most useful when
it points at one of these surfaces:

1. **The Git diff.** File paths, branch refs, and commit SHAs come
   from the user's checkout and are treated as **untrusted** —
   filenames may contain shell metacharacters or control chars, a
   ref may be hostile, and the diff may be malicious. The
   `LogSanitizer` class is the boundary at which untrusted strings
   become loggable. Path traversal protection lives in
   `SourceFileScanner` and `PathToClassMapper`. Reports about log
   forgery, path traversal, command injection, or untrusted-input
   crashes are in scope.
2. **The Java source AST.** JavaParser runs on every `.java` file
   under the configured source dirs. Reports about JavaParser
   crashing the plugin on malformed input, or about a parse
   failure causing the plugin to silently miss tests, are in scope.
3. **The Gradle build script.** The `affectedTests { ... }` DSL is
   trusted — the plugin trusts the project that applied it. A
   report saying "I configured `outOfScopeSourceDirs = '/etc'` and
   nothing happened" is not in scope; that surface is intentional.

Out of scope:

- Vulnerabilities in third-party dependencies that the plugin
  does not surface to the adopter (`./gradlew :affected-tests-gradle:dependencies`
  for the full list). Report those upstream — Dependabot will
  pick up published advisories automatically.
- Misconfiguration on the consumer side (over-broad
  `outOfScopeSourceDirs`, `mode = 'strict'` escalating in CI, etc.)
  — those are operability bugs, not security bugs, and belong in
  a regular issue.
- Anything that requires the attacker to already own a write path
  into the consumer's repo — at that point the diff is the least
  of their problems.
